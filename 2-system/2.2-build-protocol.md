# 2.2 — The build protocol

> **How you safely extend the system.** Compact and hard. Every session that builds or changes something in
> the system follows this. The goal: never "finished," but always manageable — no monster of technical debt.
> This is the System role's working standard (see [2.1](2.1-system-role.md)).

## The one rule
A system change is **done** only when it is (1) classified, (2) in its canonical home, (3) in the registry,
and (4) — if it's a process — watched. Otherwise it doesn't exist.

> **Registry = index, NOT the documentation.** The docs live *with* the component (the skill folder, the
> process folder, the repo, infra). The registry only points at them. A building session needs both: the
> index (where) + the docs (how).

## 1. Determine the kind → the home
| Kind | = | Home |
|---|---|---|
| **Kernel** | who you are / the hardest rules | the core |
| **Skill** | on-demand playbook (how to do X) | `skills/<slug>/` |
| **Process** | runs automatically (schedule/event) | `processes/<name>/` + a runner |
| **Knowledge** | facts / things to look up | your knowledge store + search |
| **State** | where we stand | databases or per-(sub)project state |
| **Technical** | how it's built (code) | a repo + README |
| **Infra** | machines, network, access | `infra/` — topology + why + runbooks |

**Skill or process? Deciding test:** does a runner/schedule/event start it **without a human** → process. Must
it be **called** (a human or an on-demand call) → **skill**, even if it "automates"/"is autonomous" or has
server tooling. No runner = no process.

## 2. Definition of Done — no "done" without this
- **Skill:** `SKILL.md` (with a sharp `description`) + `rulebook.md` + a registry entry. No status tracker in
  the folder — status lives in the registry; the folder = `SKILL.md` + `rulebook.md` (+ optional reference
  docs).
- **Process:** skill + rulebook + runner/schedule + a **health-check** + a **ping-on-failure** + a registry
  entry + a last-run log.
- **Knowledge:** in your knowledge store (searchable). Not in version-controlled skills.
- **State:** in a database or, per (sub)project, a state folder with all generated docs. The tracker = status
  + index of the folder; rulebook/memory/notes alongside. Loading = **explicit** (no native cascade): core →
  project tracker → subproject tracker. Fixed casing, no duplicate files.
- **Kernel / rule:** short, in the core. Not scattered across loose docs.
- **Technical:** in a repo with a README.

## 3. Always (on every change)
- **Update the registry** ([2.4](2.4-registry.md)): what · where · status. It's the **index** — resolve paths
  here, **never hardcode** a path.
- **Log** to your history/milestone store (the work, its phase from the fixed list, where the artifact lives).
- **No secrets** in version control (a pre-commit check guards this).
- **1:1 before upgrade** on migrations; the original stays; the maintainer reviews.

## 4. Self-learning — nothing evaporates
A correction → immediately a rule in the `rulebook.md` of that skill/process: `- [YYYY-MM-DD] <rule> (re:
<trigger>)`. The chat is not memory; the rulebook is.

## 5. The gates enforce this (see [2.3](2.3-monitoring.md))
- skill without a `rulebook.md` or registry entry → flag.
- process without a health-check → flag.
- registry/doc out of sync or a dangling reference → flag.
- a lifecycle phase outside the fixed list → flag.
- a secret in a commit → block.

## Terms
- **Canonical home** = the one right place for a thing; elsewhere is at most a copy. One source, no double
  truth.
- **Registry** = the table of contents/map: what · where · status. No code/tool explanation.

## Documentation discipline — living sources, NOT static lists
Never duplicate dynamic data as a static list (it rots). Point to the living source; only the registry is a
manual index (updating it is a hard step on every build; a governance check catches drift).

> Keep this protocol short. If it grows too long or complex itself → that's debt too. Prune.
