# 2.3 — Monitoring (the principle)

> **Tier 2 — principle, not implementation.** This doc ships the *why* and the *pattern* of self-watching, not
> scripts, schedulers, or addresses. A system that extends itself needs gates so nothing silently rots — but
> the concrete gate is something you (and Claude Code) build per setup. One anonymized example below shows the
> shape; copy the pattern, not code.

## Why continuous self-watching
The build protocol's "one rule" is only as good as its enforcement. Left to goodwill, rulebooks go unwritten,
the registry drifts, processes fail silently, and the system slowly becomes the monster it was meant to avoid.
**Gates make the discipline self-enforcing** — they catch what humans forget, so the system keeps its own
house in order: health, governance, self-healing, stuck/wedge detection.

## The hard routing rule
**An alert never goes *only* to the user.** A finding routes to **someone or something that can act on it** —
a maintainer agent, a queue, an inbox that gets worked — so problems get *handled*, not just signaled. A
notification to a human who can't fix it is fine *in addition*, never *instead*.

## Trust the signal — watch reliably, or not at all
A monitor is only useful if you can trust it. A few rules keep it honest:
- **Watch on reliable signals, not flaky heuristics.** Some states can't be detected by passively peeking
  (e.g. "is this session reachable right now?"). Detecting on a guess produces **false alarms**, and false
  alarms are worse than none: they train everyone to ignore the dashboard, so the *one real* problem hides in
  the noise. Prefer a **literal, unambiguous signal** (an explicit "failed" string, a positive "connected"
  marker) over inference. If a state genuinely can't be confirmed, report it as **unknown**, not as broken.
- **Never auto-*act* on a signal that can't tell "dead" from "alive."** Detecting is bad enough on a guess;
  *acting* on one is worse. Real case: an auto-watchdog treated *unsent text sitting in the prompt* as a
  "ghosted session" and **killed 6 live sessions** — because a working session with queued input is passively
  **indistinguishable** from a dead one. Passive ghost-detection is unreliable, and auto-reviving on a fuzzy
  signal is more dangerous than the problem it solves. If you can't distinguish the states, **surface it for a
  human — don't auto-act.**
- **A self-healing action and its alarm must not share a threshold.** If you *renew when < 2 days left* and
  *alarm when < 2 days left*, the gap between a check and its action cries wolf. Put the action threshold
  **safely above** the alarm threshold (e.g. renew at 3 days, alarm at 1) so the alarm only fires on a *real* failure.
- **Debounce transient states.** A thing that is briefly "starting"/"connecting" is normal; only flag it once
  it's been stuck **across two checks**. One bad sample is noise; a persistent one is a finding.
- **A too-strict confirmation is worse than none — it lies with authority.** A recovery that actually *works*
  can be hidden by a broken success-check that only accepts one strict form of the signal. Real case: a
  session reconnected fine every time, but the confirmation searched for the *expanded* "active" status string
  while a quiet, healthy session shows the *collapsed* indicator → every check reported "not confirmed" →
  people were sent to the backup route for nothing and formed a days-long "this is unrecoverable" belief — a
  pure measurement error. **Confirm on the real signal in *all* its forms (including the minimal/collapsed),
  not one variant.**

A monitor that cries wolf gets muted, and a muted monitor is the silent failure you were trying to prevent.

## Anchor discipline in the harness, not in memory
The deepest reason gates exist: **a rule the model is told to "remember" gets compacted away or forgotten
exactly when it matters.** So the durable way to enforce critical discipline is to **mechanize it as a
deterministic hook in the harness**, not to add another line to the context and hope. The recurring move:
every time you notice *"the agent forgot X after a compaction,"* the fix is **build a hook — don't add another
rule.** (This is the enforcement arm of [1.4 context management](../1-core/1.4-context-management.md); the
honest limit there still holds — a hook can fire reliably, but it can't *force* the model to reason.)

Self-healing checks that belong here, all routed to an actor that can act (never only the user):
- **Idle roll-up check** — when an agent sits idle for a while, nudge it: *"is everything logged per the
  conventions?"* — so progress reaches durable storage before the next compaction, not after it's lost.
- **Post-compaction re-orientation** — a session-start hook that re-injects orientation + the latest handoff,
  so an agent is automatically back up to speed without a manual kick-off.

**Session recovery is on request, not an auto-watchdog.** When a human notices a dead/ghosted session, the
orchestrator recovers it *with full context* (resume). Don't build a watchdog that kills-and-revives on a
passive guess — as above, that signal can't tell a live session from a dead one and *will* take out working
ones. An auto-watchdog is only safe with a signal that genuinely distinguishes the states, and passively that
signal doesn't exist. (Pairs with the [access model](2.1-system-role.md): app for convenience, terminal
backdoor as the guaranteed way in when a session needs recovering.)

## What gates typically enforce (as principles)
- A skill/process without a `rulebook.md` → **flag** (learnings would evaporate).
- A process without a health-check → **flag** (silent failure).
- Registry/doc out of sync, or a dangling reference → **flag** (drift).
- A lifecycle phase outside the fixed vocabulary → **flag** (inconsistent state).
- A secret about to be committed → **block** (hard stop).

## One golden example (anonymized — the *pattern*, not code)
A "rulebook gate":
1. **Trigger** — runs on a schedule (say, daily) and on every change to a skill/process folder.
2. **Check** — for each skill/process, does a `rulebook.md` exist *and* is there a registry entry?
3. **On failure** — it doesn't email the user and stop. It drops a finding into the **maintainer's inbox**
   ("`<component>` has no rulebook — created a stub, please confirm") *and* nudges the maintainer agent to act.
   Optionally it self-heals (creates the stub) so the gap is closed, not just reported.
4. **Result** — the discipline holds without anyone remembering to check.

> That's the whole point of Tier 2: you learn *what* a gate must guarantee and *why*, then build the real one
> for your own stack. The model ships the contract, not the plumbing.
