# 2.4 — The registry discipline

> The registry is the **one manual index** of everything you've built. It is the backbone of the
> [build protocol](2.2-build-protocol.md)'s "one rule": a change isn't done until it's *in the registry*.
> Ship the **discipline and the format** — never a filled-in registry.

## What the registry is
The single table of contents — **what · where · status** — for every built component (kernels, skills,
processes, infra, technical repos). It **points at** the docs; it is *not* the docs. The docs live with the
component (the skill folder, the process folder, the repo, infra).

## Why it exists
- **Resolve, don't hardcode.** Paths are resolved *through* the index, so nothing breaks when things move.
- **Discoverability.** Every built component is findable in one place.
- **Drift is detectable.** Because there's one index, a check can compare it against reality and flag gaps.

## The discipline
- **Updating the registry is a hard step on every build** — not optional, no corner-cutting. "Done" without a
  registry entry isn't done.
- A **governance check** compares the registry against what actually exists and flags drift or dangling
  references (see [2.3](2.3-monitoring.md)).
- The registry holds **what/where/status only** — no code, no tool explanations. Keep it lean.

## The format
A small table anyone can copy (see the [registry template](../3-execution/3.3-templates/)):

| Component | Kind | Where (canonical home) | Status | Watched? |
|---|---|---|---|---|
| `<name>` | skill / process / infra / … | `<path or repo>` | active / planned / archived | yes/no/n.a. |

> One index, no double truth. If you find yourself keeping a second list of the same thing somewhere else,
> that's drift — point at the registry instead.
