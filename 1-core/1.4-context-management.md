# 1.4 — Context management (the compaction discipline)

> **This is the heart of the model.** Everything else — the trinity, the cascade, the flywheel — only works if
> context *survives*. An agent's working context is finite and gets **compacted**: summarized, with detail
> dropped, to make room. The discipline here is what guarantees that a compaction costs you nothing. Read
> [1.1 the model](1.1-the-model.md) first; this chapter is how its memory actually holds up over time.

---

## 1. The guarantee lives in *continuous* write-back, not a last-moment sweep
**Nothing evaporates only if you write each thing to durable storage the moment it happens** — every
milestone, decision, deliverable and correction, immediately, on the right layer (the State changelog, the
roll-up, the rulebook). Do that strictly and a compaction loses nothing, because the durable layers already
hold it. Skip it and bet on a sweep "later," and a single compaction can wipe the whole session.

> So the flywheel and the layered logging from [1.2 §0/§5](1.2-mindset.md) aren't bookkeeping — they're the
> mechanism that makes context survivable. Log as you go, not at the end.

## 2. What auto-compaction does — and what it does *not*
A compaction **summarizes and drops detail** to free space. It does **not** do the intelligent **roll-up** —
walking back over the session and writing each outcome thoughtfully to the right durable layer. That roll-up
is **model work** (judgment about what mattered and where it belongs), and an automatic summarizer cannot do
it for you. Treating auto-compaction as if it were the roll-up is the classic way to lose work.

## 3. The honest limit (state it plainly — never promise false certainty)
Be straight about what you cannot guarantee:
- A hook or automation **cannot force the model** to perform the roll-up — that's reasoning, not a script.
- There is **no reliable "at X% context" trigger** you can hang a guarantee on.
- You therefore **cannot 100% guarantee** the roll-up happens *before* an auto-compaction fires.

Anyone who promises otherwise is gambling. Build for resilience, not for a certainty that doesn't exist.

## 4. What you *can* do — a layered defense
No single layer is airtight; together they make loss rare. Each is a buildable mechanism (see
[2.3 monitoring](../2-system/2.3-monitoring.md) for the enforcement pattern):
1. **Pre-compaction handoff** — a hook that, before a compaction, writes a fresh handoff (recent intent +
   current status) to a durable file.
2. **Compact instructions** — in the always-loaded config, steer what any summary must preserve: the current
   task, instructions verbatim, mistakes + their fixes, exact names/paths.
3. **Post-compaction re-orientation** — a session-start hook that re-injects the orientation + that handoff,
   so the agent is automatically back up to speed without a manual kick-off.
4. **Idle roll-up check** — when an agent sits idle for a while, remind it: *"is everything logged per the
   conventions?"*
5. **Proactive compaction** — when context runs high, deliberately run the full roll-up *yourself* before
   auto-compaction takes the choice away.

## 5. Orient before you act — and re-orient after a compaction
- **Read the canonical documentation/source first; reverse-engineer the live system only if it isn't
  documented** — and write back what you discover. Code defaults on a running machine are misleading; the doc
  + live status win. (Enforce with a hook that fires when you reach for live-system access and reminds you of
  the docs-first reflex.)
- **After a compaction, re-orient from durable sources before acting on the next request.** The orientation
  instruction is exactly the kind of thing a compaction discards — precisely when you need it — so don't rely
  on remembering to do it; enforce it.

## 6. The meta-principle — anchor discipline in the harness, not in memory
A rule the model is told to "remember" gets compacted away or forgotten *exactly when it matters most*. So
**mechanize critical discipline as deterministic hooks in the harness, rather than leaning on instructions
sitting in the context.** This is a recurring move: every time you catch yourself thinking *"the agent forgot
X after a compaction,"* the fix is **build a hook — don't add another rule.** Memory is where discipline goes
to die; the harness is where it lives.

## 7. Not all enforcement is equal — inject the content, not the instruction
Building a hook is necessary but not sufficient: **what** the hook injects decides whether it works. Injecting
an *instruction to go read something* — even a blunt *"STOP: don't start work until you've loaded your
operating rules and the current state"* — is **still ignorable**. The model can read the mandate and skip it.
The reliable mechanic is to **inject the source's *content* at the moment of need**: the text is simply
*there*, in context, with no choice to skip it.

> **Proof (not theory):** after a compaction, an agent received a harness-injected mandate that literally said
> *"STOP: don't begin work until you've loaded (1) your operating rules and (2) the root state file."* It
> ignored it — worked only from the working-directory context file the harness auto-loads *as content*, and
> dived into the task with absent/stale orientation. Mandate-injection ≠ enforcement; content-injection is
> what lands.

**The enforcement ladder (soft → hard):**
1. **A rule** in a charter/instructions file — pure model behavior, ignorable.
2. **Injecting a *mandate*** (a hook that says "go read X") — louder, but *proven still ignorable*.
3. **Injecting the *content*** (a hook that puts X's text in context) — reliable; it's just there. ← the sweet spot
4. **A hard gate** — block the action until a detectable precondition is true (e.g. refuse the tool call) — forces the order.
5. **Deterministic orchestration** — separate program steps; step 2 can't run without step 1's output — hardest.

Level 2 is proven insufficient; **level 3 is the cheapest reliable fix**; levels 4–5 are backstops.

Two places this pays off:
- **On session restart / compaction:** inject *as content* the agent's identity + its operating rules (read
  live from source, not a stale copy) + the root state of what it's working on. Not "go read" — the text
  itself, present. (This is how §5's re-orient actually gets enforced.)
- **On a reference into a hierarchy:** inject the **full ancestor chain** of state files (root → … → leaf) as
  content — the [cascade](1.1-the-model.md) (1.1 §5) turned from a reading *discipline* into an automatic
  **traversal**: walk the tree, inject each layer. Don't rely on the agent to walk it itself.

> This sharpens the honest limit in §3:
> content-injection guarantees the right material is *present*; it still can't guarantee the model *reasons*
> well on it (the roll-up is judgment). But present-and-maybe-skipped-by-a-bad-agent still beats
> instructed-and-reliably-ignored. Make the inputs unavoidable; backstop the reasoning with a gate.

> **Why this is the core of the model:** context management *is* the product. A system that learns your
> business is worthless if it forgets what it learned the moment its context fills up. The trinity stores it,
> the flywheel grows it — and this discipline is what carries it across every compaction in between.
