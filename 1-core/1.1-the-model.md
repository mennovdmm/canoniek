# 1.1 — The model

> **What this is.** One authoritative, self-contained description of the model behind Canon. Read this and you
> know the structure and the mechanics without opening anything else. Everything here is the *schema* — the
> universal structure. Your *content* (who you are, your clients, your work) is yours to fill in and lives
> outside this repo (see [SETUP.md](../SETUP.md) and [1.3](1.3-identity-placeholders.md)).

---

## 1. In one paragraph

The model is **a trained brain plus the discipline by which it trains itself.** The value isn't "ready-made
agents" — it's a strict, universal structure that's the same everywhere, a self-learning loop (corrections
become rules), and a fixed way the system extends itself (version control as the source of truth, everything
watched, nothing evaporates). The core principle: **ship the schema, not the data** — the *structure* is
universal (you share it), the *content* is personal (you don't). The promise to the user: *it learns your
business.*

---

## 2. The building block — the three-part unit (the trinity)

**Every node in the system is the same memory unit with three fixed parts.** The business, a client, a
project, a skill: identically built. Only the *label* specializes per layer — that's the unit: *one thing to
build, to govern, and to learn from.*

The three parts line up with the three memory types the AI field already names:

| Part (base term) | Market term | What it is | Changes |
|---|---|---|---|
| **1 · Charter** (constitution) | *semantic / core memory* | who · why · the rules (identity) | rarely |
| **2 · State** (working layer) | *working + episodic memory* | what's where & how-now **+ the dated history** | constantly |
| **3 · Rulebook** (learnings) | *procedural memory* | the rules built up from corrections — the flywheel | grows |

The **Rulebook is the third part *and* the motor under the other two**: every correction lands here, on the
right layer. That's why "learnings" never feels like an equal third box — it's the learning layer that runs
*through* all parts (see §5, the flywheel).

> **Note on "State".** "State" here means the *working-layer part within each block*. At a system level the
> same word also names a component *category* ("where do we stand" → databases / state files). Same concept at
> two levels: the State category is exactly what the State part of each block concretely contains (see §7).

## 3. The palette & the per-layer recipe

### 3a. The palette — four families
The parts come from a small, fixed palette of **four families**:

1. **Constitution** — identity, purpose, the rules (= the Charter part).
2. **Skill** — the repeatable way of working (a capability).
3. **Work** — what's-where & how-now + changelog (= the State part).
4. **Learnings / rulebook** — the flywheel; lives *in every layer*.

> **Reconciling 3 vs 4:** each block has **three** ingredients (Charter = Constitution · State = Work ·
> Rulebook = Learnings). The fourth family, **Skill**, is **not an ingredient inside every block** but its own
> *block type* in the general layer: a skill is itself a block (Charter = the way of working / `SKILL.md` ·
> State = minimal/stateless · Rulebook = its `rulebook.md`). So the trinity (3, per block) and the palette
> (4, system-wide) stay consistent.

### 3b. The recipe per layer (same three parts, different label)
The base term is fixed; in parentheses, what it means on that layer:

| Layer | **Charter** (semantic) | **State** (episodic/working) | **Rulebook** (procedural) |
|---|---|---|---|
| **System** | the constitution (the core) | infra · gates · status | system rulebooks |
| **Skill** | the procedure (`SKILL.md`) | — (stateless) | `rulebook.md` |
| **Client** | strategy · DNA (the registry) | changelog / work in progress | conventions + a copy rulebook |
| **Project** | goal · the brief (inherits from client) | status · changelog · decisions (ADR) | `rulebook.md` |
| **Subproject** | the task (inherits) | status · changelog | `rulebook.md` |

**The client registry** (a client's Charter) is **not a fill-in exercise but a derivative**: the sum of the
intake/onboarding + *all* of that client's projects, summarized automatically by the AI into a few generic
blocks (who · context · core/DNA verbatim · origin · rolled-up status · agreements + learnings). Every
business has its own onboarding, but there is always *an* onboarding.

## 4. The layers as a tree

The blocks stack into a **tree** with a horizontal split:

**General layer** — shared, reusable across *all* clients:
- **Core** (layer 1) — the always-loaded constitution; the only command layer.
- **Skills** (Execution, layer 3) — shared capabilities, each `SKILL.md` + `rulebook.md`.
- **Processes** (Execution) — anything that runs *automatically* (runner/schedule) = skill + rulebook +
  self-learning + a runner. *Deciding test:* does a runner/event start it without a human → **process**; must
  it be called → **skill**.
- **Infra** (System, layer 2) — the engine room: machines, network, tooling, gates. Not a knowledge block —
  the layer underneath that makes everything run.

**Content layer** — per relationship, **× ∞** (infinitely many clients × projects × subprojects):
- **Client → Project → Subproject**, a tree with many siblings per layer (one-to-many).

> **Three layers, formally:** **Core** (always loaded, lean) · **System** (infra + operation) · **Execution**
> (skills/processes, lazy-loaded). Content lives **outside the repo**. This is the physical map: **the system
> is version-controlled** (source of truth); **knowledge + content state live in external storage**
> (searchable); **hard state lives in databases**. *Which* tools you use for each is yours to choose.

## 5. The two movements — cascade ↓ and flywheel ↑

### 5a. Cascade ↓ — context flows down (a RULE, not an automatism)
An agent working on a leaf loads the whole line above it into its context: core → client root → project →
subproject. Strategy is thereby *structurally* tied to execution.

> **Crucial:** this is **not a native auto-cascade.** Most tools only auto-load a context file sitting *right
> in* a folder, not one nested inside a state subfolder. So the cascade is a **discipline / reading
> obligation** that the agent performs — prescribed in the core. **The agent reads its own branch, not the
> side branches** (loading other clients/projects collapses the context). Because it's a rule and not
> enforced by the runtime, it can be skipped — a known open point (enforcement).

### 5b. Flywheel ↑ — corrections flow up (the self-learning loop)
Every correction becomes a **permanent rule in the `rulebook.md` of the right layer** — not in a chat (which
forgets). The fixed pattern:

```
run → user corrects → learning (fixed format: - [date] <rule> (re: <trigger>))
     → appended to the rulebook → next run loads the rulebook → better
```

A **gate watches** that corrections actually land in the rulebook (otherwise a learning is lost) and flags a
skill/process *without* a rulebook. This is "always be learning": *unlogged feedback is waste; logged
feedback is an investment.* Known risk to manage: **rulebook poisoning** (contradictory/stale learnings) →
curation/pruning needed.

## 6. The agent as cursor

A **signal comes in** — a message, a question, a transcript, a chat. That determines **which branch** the
agent opens. The agent walks that one branch and **fills its context window slice by slice from every layer**:
the system rules, a **skill** (picked from the general layer), the client strategy, the project goal, the
task. The **other clients/projects stay closed** — so context doesn't bloat.

Two movements, from the agent: **Read & Write** along the branch.
- **Read** ↑ — assembles the branch (leaf → root): *enough* context to do the task, **not the whole tree**.
- **Write** ↑ — writes progress (state rolls up) and learnings (→ the rulebooks) = the flywheel.

Result: **always enough context to act, always within the frame of the client — never the whole world.**

> **One core, two roles.** There is **one shared core** that every session loads — the machinery beneath
> (skills + processes + rulebooks + gates) is shared. The difference isn't a second core but a **role
> overlay**:
> - **The agent role** (default, every session): gather as much relevant context as possible (or identify
>   where to get it) to solve the user's request, within the frame of a client/topic/task; keep going, don't
>   stop at every step; search-first (search, cite the source, *then* advise).
> - **The System role** (the maintainer session): on top of the core, loads the **builder overlay** — classify
>   → build → test → record + commit — with a governor (build in service of real, recurring friction, not for
>   the sake of building).

## 6A. The operational layers — how the system actually runs

The model above is the *structure*; these layers are *how it runs*. Each is a **universal pattern** (the
schema). The concrete tools that fill each slot are the *instance* — yours to choose, not part of what you
ship.

**a. Input / signals — how work arrives.** Work arrives through channels, and the incoming item determines
which branch the agent opens: a task · a question · a message · voice/transcript · an intake. These carry
**instructions + context + sentiment** and feed the **State** of the right node.

**b. Comms routing — processes that deliver incoming work to the right agent.** A *process* (automatic) reads
incoming comms, determines client/topic, and drops it into the **inbox** of the right agent —
**draft-first, never sending on its own.**

**c. Agent runtime — the hands that execute the model.**
- **One standing agent per client**, hard (no sprawl; exceptions only deliberate + registered): each runs its
  own branch. Plus role agents (the System maintainer; a product agent; etc.).
- **Sessions:** each agent runs as a continuous session on an always-on host.
- **Inboxes:** each agent has an inbox; work is delivered there; a handled item is removed.
- **Reachability:** each agent session is reachable/controllable from any device.

**d. Watching + alert routing — gates never go only to the user.** Checks/gates run continuously (health,
governance, self-healing, stuck detection). **Hard rule:** an alert never goes *only* to the user — it goes to
**someone or something that can act**, so problems get *handled*, not just signaled.

**e. Substrate / infra — what it runs on.** The model needs a substrate: a **system of record** (versioned), a
**knowledge store** (searchable), **hard state** (databases), an **always-on compute host**, and **reachable
devices** over a private network.

> **Schema vs instance:** the *patterns* above — input channels, comms-routing processes, one-agent-per-stream
> with an inbox + reachability, gates that route to an *acting* actor, and a version-control + knowledge +
> database + always-on-host substrate — are **universal: they belong to the model.** The concrete *names* of
> the tools are an *instance* — you don't ship those; you fill in your own channels and tools.

## 7. The temporal dimension — what + when

Context is not only **what** (the cascade) but **when**. The State part of every block is **dated**, and you
weigh each source by date (recency): *the latest meeting is only leading if it's recent* — otherwise you
reconcile across sources by date.

**Take State database-first**, not by reading every folder:
- **History / milestones** — who did what when; the fast cross-project cross-section consulted first.
- **Open items** — what's outstanding.
- **Owner context** — situation per client (commercial, contract, sensitivities).
- **State trackers** (per project) — the *deep* detail when you zoom in.
- **Scratch memory** (local) — a disposable scratchpad per workspace.

Order: **database → tracker → folders.** A controlled vocabulary for the lifecycle phase keeps status
consistent; a gate rejects a phase outside the fixed list. **Logging is layered** (same structure as reading):
detail in the subproject changelog · roll-up in the project root · always the history/milestone log.

## 8. The core principle — ship the schema, not the data

> **Structure is universal. Content is personal.**

What gets productized/shared is the **format**, not the content:

| Universal — **ship this** (the schema) | Per-instance — **don't ship this** (the data) |
|---|---|
| The format of a skill (`SKILL.md` + rulebook) | The concrete skills |
| The format of a process / the gates framework | The concrete processes / which gates someone sets |
| The memory schema + the trinity (Charter/State/Rulebook) | The memories / the filled-in blocks |
| The core format + the System overlay + the bootstrap interview | The user's filled-in core |
| The registry discipline | What's in it |

**Consequence:** dynamic data always lives **externally** (storage / database / CRM / version control), never
in the structure repo. The repo = logic + structure (versioned); external = data (per user). That makes an
instance **disposable / rebuildable**. It also answers the objection *"many rulebooks are for someone else's
use case"*: those **don't ship — the format ships**, and you fill it in yourself.

---

## 9. Reconciliations called out explicitly

1. **The trinity is newer than the taxonomy.** The taxonomy says *which kinds of components exist and where
   they live*; the trinity says *how each block is internally built* and *that it's recursively identical*.
   They don't contradict.
2. **"State" — two levels, same concept** (see §2 note): component category ⇄ the State part of each block.
3. **3 ingredients vs 4 families** (see §3a note): Skill is a block *type* in the general layer, not an
   ingredient inside every block.
4. **Cascade = explicit rule, not native automatism** (see §5a): a reading obligation, not runtime-enforced —
   a known enforcement open point.
